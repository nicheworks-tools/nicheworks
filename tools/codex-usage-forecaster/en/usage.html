What is this?

When you only see usage percentage (%) but not the absolute quota, this tool estimates your burn rate from logs and predicts depletion ETA for the 5-hour and weekly windows.

It does not try to reveal the absolute cap (often unpublished/variable). It stays useful even when caps change.

Why burn-rate?

  • Absolute caps may be unpublished or vary by plan/model/system load
  • Same work can “cost” different % depending on settings
  • Reset timing is hard to feel (per-window rolling behavior)

What you log

  • 5-hour usage (%) (optional)
  • Weekly usage (%) (optional)
  • Mode: cloud / cli / local / other
  • Model/setting: e.g. gpt-5.2 / mini / high (optional note)
  • Note: what you did

You can save with only one of the two percentages. Both empty is not allowed.

Forecast logic (current minimal robust version)

  1. Use only positive deltas of % between logs (ignore resets/zeros)
  2. Compute burn rate:
    • 5-hour: % per hour
    • Weekly: % per day
  3. Use median of recent deltas (up to 5) for stability
  4. ETA = (100 - current%) / burn rate

Privacy / Data

  • Stored in browser localStorage (this device/browser only)
  • No uploads / no server sync
  • Use Export/Import for migration

Troubleshooting

  • ETA says “need more logs”: you need 2+ logs for that window
  • Weird burn rate: a noisy log may exist—delete it
  • Reset caused % to drop: normal—add 2 logs after reset
  • Data missing on another device: use Export/Import
