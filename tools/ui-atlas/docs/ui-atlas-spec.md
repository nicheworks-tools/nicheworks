# UI Atlas

**製品区分:** 有料化可能な実用Webツール
**運営名義:** Niche Works
**製品カテゴリ:** UI辞典 / UI比較辞典 / AI依頼補助辞典
**対応言語:** 日本語優先、英語は将来拡張可能だが初期完成形では日本語中心
**対象:** Web制作初心者、ノーコード利用者、AIでUI生成する人、フロントエンド初中級者、ディレクター、デザイナー補助層

---

## 1. 製品の定義

Niche Works UI Atlas は、Web制作やUI設計で頻出するUIパターンを、**名称・見た目・用途・非推奨用途・類似UIとの差分・AIへの依頼例・実装メモ**まで一体化して整理した実用辞典である。

単なるUIギャラリーではない。
単なる用語辞典でもない。
単なるAIプロンプト集でもない。

この製品の価値は、次の7つを**1項目の中で完結**させることにある。

1. そのUIの名前が分かる
2. 何のためのUIか分かる
3. どんな場面で使うべきか分かる
4. どんな場面では使うべきでないか分かる
5. 似たUIと何が違うか分かる
6. AIにどう依頼すればよいか分かる
7. 実装時に何を意識するか分かる

---

## 2. 目標

### 2-1. ユーザー目標

ユーザーはこのサイトを使って、次のような状態になれること。

* 「あのUIの名前何だっけ」を解決できる
* モーダルとボトムシートの違いのような**似ているUIの差分**を理解できる
* AIにUI修正やUI生成を依頼する時に、**曖昧ではない言い方**ができる
* UI採用判断の失敗を減らせる
* 実装時の最低限の注意点を把握できる
* 比較しながら選べる
* 辞典として読むだけでなく、作業に直結して使える

### 2-2. 事業目標

Niche Works にとっては、以下を狙う。

* 検索流入が取りやすいロングテール型の資産
* SNSでスクショ共有されやすい一覧系ツール
* AI時代に合った「辞典 + 実用品」ポジションの確立
* 無料流入を取りつつ、有料で深い比較・コピー・保存を売る
* 他のNiche Works製品への横展開の起点

---

## 3. 非目標

この製品でやらないことも明確にする。

* Figmaファイル配布サイトにはしない
* コードスニペット集が主役ではない
* UIテンプレ販売サイトにはしない
* ノーコードビルダーそのものにはしない
* 一般的なデザイン学習講座サイトにはしない
* DribbbleやBehanceのような雰囲気参考集にはしない

この製品はあくまで、**辞典としての網羅性と、実務での使いやすさを両立すること**に集中する。

## 9. 各UI項目ページの構成

各項目ページはこの順で固定する。
読む順番が毎回同じであることが、辞典としての強さになる。

### 9-1. ヘッダー領域

* UI名（日本語）
* 英語名
* 別名
* 所属カテゴリ
* 難易度タグ
* モバイル相性タグ
* AI向きタグ

### 9-2. 一言説明

20〜40文字程度で何かを言う。

### 9-3. ミニプレビュー

静的画像ではなく、簡易インタラクティブ見本が原則。
ただし重い実装は禁止。
「見たらどういうUIかすぐ分かる」程度の軽量再現を行う。

### 9-4. 何に向いているか

箇条書きではなく、読みやすい短文で3〜5行。

### 9-5. 向いていない場面

ここは必須。
これがない辞典は浅い。

### 9-6. 似たUIとの違い

比較表を入れる。

例:

| 比較対象         | 主な違い        | どちらを選ぶべきか          |
| ------------ | ----------- | ------------------ |
| Modal        | 中央表示、強制感が強い | 重要確認ならModal        |
| Bottom Sheet | 下部表示、モバイル向き | 補助設定ならBottom Sheet |

### 9-7. AIにそのまま使える依頼文

用途別に複数出す。

* 基本版
* モバイル版
* アクセシビリティ配慮版
* リファクタ版

### 9-8. 実装メモ

* 何の状態が必要か
* どの入力で開閉するか
* キーボード操作が必要か
* focus trapが必要か
* body scroll lockが必要か
* aria属性で何を意識すべきか

### 9-9. よくある誤用

例:

* FAQなのにPopoverで出そうとする
* 破壊的確認をBottom Sheetで済ませる
* モバイルで巨大なMega menuを使う

### 9-10. 関連項目

内部回遊を強くする。

## 18. デザイン仕様

この製品自体のUIが弱いと説得力が落ちる。
ただし、凝りすぎると辞典の可読性が落ちる。

### 18-1. デザイン方針

* 情報が主役
* 装飾より判別性
* 一覧の視認性を最優先
* 比較しやすい余白
* ミニデモが埋もれない
* 長文でも読み疲れしにくい

### 18-2. トーン

Niche Worksらしく、

* 実用
* 静か
* 軽い
* 過剰に派手ではない
* ツール感が強い

### 18-3. カラー方針

* ベースはニュートラル
* カテゴリ色は薄く差をつける
* CTA色は1色に集約
* 比較差分の強調色を明確化

## 29. 無料と有料の境界設計

この製品は無料範囲が弱いと流入を失うし、有料範囲が弱いと稼げない。
境界はこうする。

### 無料で十分できること

* 名前を知る
* 基本用途を知る
* 簡単な違いを知る
* 短いAI依頼文を使う

### 有料で速く深くできること

* 複数候補の比較
* 迷いどころの整理
* 詳細な依頼文コピー
* 保存
* 実装判断の補助
* 再利用

つまり、有料は**知識課金ではなく作業効率課金**にする。

## 31. 失敗条件

* 項目数が少ない
* 似たUIの違いが薄い
* AI依頼文が1本しかない
* 実装メモが浅い
* 検索が正式名称しか拾えない
* モバイルで比較が見にくい
* 辞典なのに更新が止まる
* 有料の理由が曖昧

## 32. この製品で金を取れる理由

最後に、なぜ金が取れるかを明確にする。

この製品は、単なる知識サイトではなく、

* **探す時間**
* **迷う時間**
* **AIに言い直す時間**
* **似たUIを比較する時間**
* **前に見た情報を探し直す時間**

を削る。

つまり売るものはUI用語ではなく、
**UI判断とAI依頼の時間短縮**です。

この設計なら、無料で人を集めつつ、有料で十分に課金理由を作れます。
